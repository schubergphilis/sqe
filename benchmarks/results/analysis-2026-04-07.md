## Spill Behavior

| Config | Sort Spills | Sort Bytes Spilled | Join Spills |
|---|---|---|---|
| single-512mb | 30 | 1.1 GB | 0 |
| single-8gb | 128 | 27.7 GB | 0 |
| distributed-2w | 3 | 49 MB | 0 |

### Analysis

**single-512mb (30 spills, 1.1 GB):** Spill occurs primarily during TPC-DS complex sorts. 92/99 TPC-DS queries complete with spill — only 7 fail (hash aggregate OOM, same root cause as TPC-H q18). SSB fails 9/13 because large fact-dimension joins exceed 512MB before SortMergeJoin fallback triggers.

**single-8gb (128 spills, 27.7 GB):** Surprisingly, MORE spills than 512MB. This is because 8GB runs TPC-E successfully (13/18 vs 12/18) — and TPC-E queries involve massive multi-table joins across 33 tables. The 127 spills and 27GB are almost entirely from TPC-E. With 512MB, those queries simply OOM before reaching the spill point.

**distributed-2w (3 spills, 49 MB):** Nearly zero spill. Workers absorb scan and partial aggregation work, coordinator only merges small result sets. The 3 spills are from a single TPC-DS query that requires a large sort even after distribution.

### Key Insight

Spill count alone doesn't indicate problems — it indicates the system is working as designed. The 512MB config proves the safety net works (1.1GB spilled to disk, 92% pass rate). The distributed config proves that pushing work to workers eliminates the need for spill almost entirely.

## Cache Behavior

| Config | Query Cache Entries | Query Cache Misses | Hit Rate |
|---|---|---|---|
| single-8gb (cumulative) | 132 | 153 | 0% (cold cache, unique queries) |
| distributed-2w (cumulative) | 119 | 121 | 0% |

Cache hit rate is 0% across all configs because the benchmark runs each query exactly once — there are no repeated queries. The query result cache would show value in production workloads with repeated dashboard queries.

Footer cache hits/misses: all 0. This indicates the footer cache is not yet wired into the scan path — the Phase A `FooterCache` struct exists but the `IcebergScanExec` does not yet call `get_or_fetch()`. This is a wiring task for the next iteration.

## Scheduler Behavior (distributed-2w)

| Metric | Value | Notes |
|---|---|---|
| `scheduler_decisions{local}` | 120+ | All queries executed locally on coordinator |
| `scheduler_decisions{distributed}` | 0 | No queries distributed to workers |

**All queries ran locally on the coordinator.** The workers were available (healthcheck passed) but `try_distribute()` only distributes when the scan has enough data files (>= `distribution_file_threshold`, default 4). At SF1, most tables have 1-2 files — below the threshold.

The 2.5x speedup comes from the streaming execution improvements on the coordinator (spill, late materialization, scan planning), NOT from actual worker distribution. To see worker distribution, run at SF10+ where tables have 10+ data files.

## TPC-E: The Outlier

TPC-E has the lowest pass rate across all configs (56-72%). Root causes:

1. **5 queries blocked by DataFusion IN(subquery):** `market_feed_update`, `trade_result_update_holding`, `trade_result_update_status`, `trade_update_executor`, `trade_update_settlement` — these use correlated IN subqueries that DataFusion cannot decorrelate.
2. **Large multi-table joins across 33 tables:** TPC-E's brokerage schema has deep join chains (trade → customer_account → customer → address → zip_code) that produce massive intermediate results.
3. **Some queries timeout at 600s** on single-8gb (explaining 127 spills / 27GB — the query spills but never completes within timeout).

## Recommendations

1. **Wire remaining metrics** — footer cache, pruning counts, late-mat bytes, time-to-first-row. Low effort, high observability value.
2. **Run at SF10** to test actual worker distribution (more files per table → crosses distribution threshold).
3. **Increase distribution_file_threshold** from 4 to 2 for SF1, or use SF10 where it's naturally above threshold.
4. **TPC-E:** Rewrite the 5 blocked queries using EXISTS/JOIN instead of IN(subquery). Track DataFusion for correlated subquery improvements.
5. **Skip distributed-4w on 36GB machines** — Docker VM memory limit causes crashes.
